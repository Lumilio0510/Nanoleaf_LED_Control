## Architecture

This is an **Electron** desktop app for controlling LED lights over HTTP, with an AI chat feature that generates LED effect "Skills" from natural language via OpenAI/Ollama.

### Three-tier Electron structure

```
src/main/        Main process (Node.js)
src/main/preload.ts  Context bridge — exposes window.electronAPI to renderer
src/renderer/    Renderer process (React + MUI)
src/shared/      Types and IPC channel constants shared by both
```

Build tool: **electron-vite** — outputs to `out/main/`, `out/preload/`, `out/renderer/`.

### Main process

- **`main.ts`** — App entry. Creates `BrowserWindow`, calls `registerHandlers()`.
- **`ipc-handlers.ts`** — Central routing: all `ipcMain.handle`/`ipcMain.on` registrations. Each handler delegates to a service. This is the switchboard between renderer and services.
- **`agent.service.ts`** — Chat orchestration. Builds a system prompt (LED API doc + Skill JSON schema), calls LLM via adapter, extracts Skill JSON from LLM response via regex.
- **`llm/`** — `LLMAdapter` interface with `chat` and `chatStream`. Two implementations: `openai.adapter.ts` (OpenAI-compatible API) and `ollama.adapter.ts` (Ollama local API). Selected by `getAdapter(config)` based on user's LLM config.
- **`skill.service.ts`** — Skill CRUD. Skills stored as individual JSON files in `userData/data/skills/`.
- **`skill-executor.ts`** — Resolves Skill params against user values, renders `{{params.xxx}}` template in `bodyTemplate`, parses `METHOD /path` endpoint, calls `sendRequest()`.
- **`led-api.service.ts`** — HTTP client for the physical LED device's REST API (`/light/power`, `/light/brightness`, `/light/color`, `/effect/:id`, `/effect/list`).
- **`device.service.ts`** — Device lifecycle: LAN scan (ping all IPs on subnet at port 8080), connect/disconnect with ping-based heartbeat (10s interval, 3 failures = disconnect). Status changes broadcast to all renderer windows.
- **`storage.ts`** — Thin JSON file persistence in `app.getPath('userData')/data/`. All data is file-based, no database.

### Renderer process (React)

- **`main.tsx`** — Entry, wraps App in MUI `ThemeProvider`.
- **`App.tsx`** — Root layout: sidebar + main content area + status bar. Four pages: `control`, `skills`, `agent`, `settings`, switched via state.
- **`api.ts`** — Typed wrapper around `window.electronAPI` (the contextBridge API). Re-exports types from `shared/types.ts`.
- **`theme.ts`** — MUI v9 theme with green primary (#10B981), custom overrides for buttons, cards, inputs, etc.
- **`hooks/useChat.ts`** — Chat state management: sessions list, current session messages, streaming text accumulation. Uses `onStreamChunk` listener with `__DONE__` sentinel.
- **`hooks/useDevices.ts`** — `useDeviceStatus` (reactive via `onDeviceStatusChange`), `useSavedDevices`, `useScan`, `useEffects`.
- **`hooks/useSkills.ts`** — Skills list with refresh/save/delete/execute.

### IPC & data flow

- **Invoke (request-response)**: Renderer → `ipcRenderer.invoke(channel, ...args)` → `ipcMain.handle(channel, handler)` → return value. Used for most operations.
- **Send (push)**: Used for streaming chat. Renderer sends `agent:chatStream` via `ipcRenderer.send`, main process pushes chunks back via `event.sender.send(agent:onStreamChunk, chunk)`. Stream ends with `__DONE__` sentinel chunk.
- **Broadcast**: Device status changes are pushed to ALL windows via `BrowserWindow.getAllWindows().forEach(win => win.webContents.send(...))`.

### Key concepts

- **Skill**: A parameterized macro mapping to an LED API call. Has metadata, params (with type/range/default), and a mapping (HTTP method/path + body template with `{{params.key}}` placeholders). Can be manually created or generated by the AI Agent.
- **Agent**: The AI assistant that generates Skills from natural language ("make a rainbow effect"). The system prompt includes the LED device's API documentation (loaded from `userData/data/api-doc.md`) and instructs the LLM to output Skill JSON.
- **Device**: A physical LED controller at `host:port`, discovered via LAN IP scan (sequential ping to all 254 IPs on the subnet at port 8080). Connection is maintained via periodic ping heartbeat, with automatic disconnect after 3 consecutive failures.
